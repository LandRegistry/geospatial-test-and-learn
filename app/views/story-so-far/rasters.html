Rasters

DRAFT CONTENT

The aim of this piece of work is to "develop and prove a method for transforming scanned spatial data into machine interpretable spatial data (digitising raster titles)"

What is the problem?

More than half of our spatial data isn't currently held in a digital format. Over 50% of HMLRs spatial data still exists in a format which is not machine interpretable, not machine readable and not able to be used with modern digital tools. This is split primarily between things which are stored on paper and things which are stored as scanned documents - essentially as useful as paper, with the exception that you can view it anywhere at any time.

This is quite a big deal for the ambition of having a geospatial register as it means over half of our spatial data cannot currently be used. To fix this, that data needs to be digitized and made to be machine readable.

What is a raster?

We are calling these file types rasters, but unfortunately they're not actual rasters, they're essentially just a picture. A raster as we've referenced it in this project is simply a digital scan of a title plan document. These are generally stored as PDFs. They represent one of 3 ways our plans data is currently stored, the others being paper plans in the File Store which have never been scanned in, and the digital plans currently held in TMF. It contains no metadata or further information about the spatial extent it indicates. That indication is only for the human eye as it is currently not machine readable.



The work in digitising rasters will transform the spatial extent into vector data to be included in the transformed dataset for the Bristol 5K. The data will be indistiguishable from the transformed TMF data, building out a robust view of the sample data.

What have we done?

Proposed method

The proposed method is to start simple and build complexity as needed. The test area of the Bristol 5K contains 10,891 plans. From these plans we identified those which are simplest to transform as they contain the least, but most important information - red edging only. The complexity will then start to be built in, looking at plans with more and more coloured spatial extents to increase the percentage plans able to be digitised.

It is unlikely every single plan will be able to digitised automatically, due to the complexity and illegibility of existing plans, even from the human eye. However, the percentage that can be digitised is really promising.

Outcomes

We identified that 84% of plans contain red edging only (9,344 with no reference extents). Because rasters are not machine readable, and it would be an enormous effort for a human to check such large quantities of plans, this is really important information which we've never had before. Understanding the range of raster types gives us a far better understanding of the problem.

This is for the Bristol 5K sample and will be validated with the 3 selected local authorities to confirm if these figures remain consistent across different geographies (rural, urban and coastal).

Red edge polygon extraction

Once this was identified we could start to test methods for extracting the red edging from those plans. We started with a computer vision approach, which looks for areas which contain colour within a certain threshold.

Originally this resulted in around 9,000 polygons being extracted from the pages - at least 1 polygon being extracted on over 90% of all red only plans in the sample area. Further work to refine this method has resulted in at least 1 polygon being extracted from every red only map page.

Computer vision based method works really well for maps with only red, but we've discovered that it doesn't generalise well to maps with multiple colours. Currently we are investigating deep learning based methods of identifying red edging where map images have more coloured extents than just red edging. This is already proving really successful.

This is a shift in moving from a computer vision solution, which is absolute, to a probabalistic method. It has a focus on object detection and image segmentation - which are used all around us everyday, such as where you blur your Teams background and it will identify where you (the object) to remain clear.

Geolocation

Once the polygon is extracted from the plan, we know the shape of the thing, it's boundaries, but we don't know where it exists in the real world. The plan image has a coordinate, but that only corresponds to its relative position on the map, not on the ground. Much like trying to put together a jigsaw puzzle, extraction gives us the shape of the piece, but we still need to figure out where it fits.

Because raster images don't contain any spatial metadata, the scale, orientation and location values which would easily give us the true alignment just don't exist. Therefore we need a method to geolocate where that polygon actually references.

To geolocate the relative position from the rest of the detail on the map, it is possible but poses it's own host of complications. To use our proposed method of starting simple, we work towards the idea that the most extracted shapes should be geolocatable in a more straightforward way.

We already hold, or have access to, a lot of different spatial data related to titles, some of which matches title extents in a lot of instances. Instead of geolocating our extracted polygons, we instead have focused on identifying where our existing spatial data matches the shape of the shape on the raster plan.

This means rather than worrying about geolocating and scaling the extracted polygon, we can instead compare this polygon to existing spatial data and replace the extracted polygon with correctly created polygons.

So far, over 90% of the red-edged only extents have been matched to an index polygon with a high degree of accuracy.
