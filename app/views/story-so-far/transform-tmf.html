Transforming TMF

DRAFT CONTENT

What is TMF?

HMLR’s spatial data is still stored in Tenet Mapping Format (TMF), a proprietary, non‑open format dating back to the 1990s. Our Computer Mapping System uses this proprietary format from Envitia Limited, creating a corporate risk due to vendor dependency and incompatibility with modern geospatial technologies.

Why transform TMF?

At present:

  • It is not widely supported
  • The format is proprietary and fragile
  • Using TMF doesn't align with open, sustainable data standards
  • Our spatial data cannot be used with external geospatial systems
  • The constraints of TMF vastly limit the opportunities for HMLR's geospatial ambitions

This legacy constraint limits automation, data usability, and interoperability. In other words, TMF is a flashing warning sign, and risk that needs to be addressed before any progress to be made on the state of HMLR's spatial data.



We need to convert this data into structured, interactive, machine-readable formats. We will then have a comprehensive repository of geospatial data related to property and ownership, accessible in multiple ways for various uses by different customers. This will unlock numerous opportunities, further enhanced by enabling technologies.

What did we do?

  • Extracted TMF data from legacy systems into a modern cloud database (Postgres).
  • Converted TMF polygons and attributes into open geospatial formats (WKT/PostGIS) using a repeatable, automated process.
  • Validated the results by comparing transformed geometries against existing CMS exports and OS mapping—showing strong alignment.
  • Built early tools and views that allow geospatial search, visualisation, and linking with register data.

What did we prove?

This worked has proved that TMF data can be transformed reliably outside existing casework systems. It has also shown that the transformed data works well in modern GIS tools such as QGIS. With the transformed data we can perform spatial searches (e.g., “find interests within 10 metres”), something not possible with TMF. The work in transforming TMF also shows that there is a viable path to linking geospatial extents with textual register entries.

Why does it matter?

The issues with TMF at present are many, and only present more risks over time. Establishing a viable method to transform TMF gives us a clear path forward in securing the use of our data in the future.

"TMF has been a major blocker to modernisation. This work shows we can move away from a proprietary, fragile format into open, sustainable data standards. "

Transforming TMF reduces corporate risk and opens opportunities for:

  • Better geospatial services
  • Enhanced Digital Register capabilities
  • Future casework tools not tied to TMF
  • Improved automation and data quality

We demonstrated that transforming TMF into open geospatial formats is feasible, accurate and repeatable. This unlocks the potential to modernise mapping systems, improve register data, and support future digital services.

Known limitations

Unhandled TMF classes

We have been investigating TMF further and it's known limitations. There are some classes in the sample data which don't map to typical geometric classes (lines, polygons, etc), these include:

  • Splines - a type we believe is specific to HMLR (curly lines with points at each end)
  • Styling - We have verified we can reproduce the styles from the title plan from the attributes in the TMF, however we have not aimed for a 100% accurate reproduction so far. There are style aspects we are now exploring:
    • Fonts
    • Text alignment
    • Varieties of hatching

Invalid geometry

We have had cases within the sample data where the resulting geometry was not “valid” according to OGC standards. We need to verify if this is due to the geometry stored within the TMF or an issue with how we transform it

Unexpected feature ids

A small number of the resulting geometries (85 out of 3.5 million) had feature ids which didn't match what we would expect from their geometry types. We don't know if this is an issue with the source data, the TMF library, or our conversion code.

Arc transform

The arc transform was complex but we limited the scope of its testing appropriately to the current "test and learn" phase of work.
